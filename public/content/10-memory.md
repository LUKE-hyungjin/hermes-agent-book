# 10. 메모리

> Hermes의 메모리는 무한 저장소가 아니라, 다음 세션에서도 꼭 알아야 할 사실만 남기는 "작고 선별된 장기 기억"입니다.

## 메모리 구조

Hermes는 세션이 끝날 때마다 모든 대화를 길게 끌고 가지 않습니다. 대신 중요한 사실만 추려 `~/.hermes/memories/` 아래 두 파일에 저장합니다.

| 파일 | 용도 | 문자 제한 |
|------|------|-----------|
| `MEMORY.md` | 프로젝트, 환경, 관습, 학습된 요령 | 약 2,200자 |
| `USER.md` | 사용자 성향, 선호, 커뮤니케이션 스타일 | 약 1,375자 |

이 두 파일은 세션 시작 시 시스템 프롬프트에 "동결된 스냅샷"으로 주입됩니다. 즉, 세션 도중 메모리를 수정해도 이미 열린 세션의 시스템 프롬프트는 바뀌지 않고, 변경 내용은 다음 세션부터 반영됩니다. 이는 프리픽스 캐시를 유지해 성능을 지키기 위한 설계입니다.

## 프롬프트에 어떻게 들어가나요?

메모리는 사람이 읽을 수 있는 간단한 블록으로 주입됩니다.

```text
MEMORY (67% — 1,474/2,200 chars)
User's project is a Rust web service at ~/code/myapi
§
This machine runs Ubuntu 22.04 and has Docker installed
§
User prefers concise responses
```

중요한 점은 세 가지입니다.

| 특징 | 의미 |
|------|------|
| 사용량 표시 | 메모리가 얼마나 찼는지 에이전트도 인식 |
| `§` 구분자 | 개별 메모리 항목 경계 유지 |
| 동결 스냅샷 | 세션 중간에는 자동 갱신되지 않음 |

## 메모리 도구 동작

Hermes는 `memory` 도구로 메모리를 다룹니다. 가능한 액션은 `add`, `replace`, `remove` 세 가지뿐이며, 별도 `read` 액션은 없습니다. 읽기는 세션 시작 시 이미 시스템 프롬프트에 포함되기 때문입니다.

```python
memory(action="add", target="user",
       content="User prefers concise Korean answers with short code examples")

memory(action="replace", target="memory",
       old_text="Ubuntu 22.04",
       content="This machine runs Ubuntu 24.04 and has Docker + Podman installed")
```

`replace`와 `remove`는 전체 문장을 다시 쓸 필요 없이, 유일하게 식별되는 짧은 부분 문자열만 `old_text`에 넣으면 됩니다. 다만 같은 문자열이 여러 항목에 들어 있으면 더 구체적으로 지정하라는 오류가 납니다.

## `memory`와 `user`의 차이

메모리를 잘 쓰려면 두 대상의 역할을 분리해야 합니다.

| 대상 | 저장할 것 | 예시 |
|------|-----------|------|
| `memory` | 환경 사실, 프로젝트 규칙, 도구 이슈, 배운 요령 | "이 저장소는 pnpm monorepo" |
| `user` | 사용자 취향, 답변 길이 선호, 민감한 포맷 취향 | "긴 서론을 싫어함" |

예를 들어 "이 서버는 Ubuntu 22.04"는 `memory`에 가야 하고, "나는 TypeScript 예제를 선호한다"는 `user`에 가야 합니다.

## 무엇을 저장해야 하나요?

Hermes는 모든 사실을 저장하는 것이 아니라, 나중에도 반복 가치가 있는 정보만 남겨야 효율적입니다.

| 저장 권장 | 저장 비권장 |
|-----------|-------------|
| 반복적으로 다시 설명하게 되는 환경 사실 | 일회성 TODO |
| 항상 유지되는 프로젝트 규칙 | 오늘만 유효한 임시 지시 |
| 장기적으로 유지되는 사용자 선호 | 채팅 한 번의 우발적 표현 |
| 실제로 검증된 작업 요령 | 추측이나 미확인 정보 |

예를 들어 "사용자는 설명이 길면 싫어한다", "이 프로젝트는 FastAPI + PostgreSQL 구조다", "이 서버는 Docker와 Podman을 모두 지원한다"는 저장할 가치가 큽니다. 반면 "오늘은 배포하지 말자" 같은 문장은 메모리보다는 현재 세션 문맥에 남는 편이 맞습니다.

:::tip
메모리는 많을수록 좋은 것이 아니라, 다음 세션 시작 시 바로 도움이 되는 정보만 남을수록 좋습니다. 꽉 차기 시작하면 새 정보를 추가하기보다 오래된 항목을 `replace`로 통합하는 편이 더 효과적입니다.
:::

## 운영 감각

메모리는 세션 복구와는 다릅니다. 세션이 "무슨 대화를 했는가"를 보존한다면, 메모리는 "앞으로도 계속 알아야 할 사실"만 요약합니다. 따라서 좋은 운용 방식은 이렇습니다.

```bash
hermes --continue
hermes chat -q "지금 기억하고 있는 사용자 선호와 프로젝트 특성을 반영해서 답해줘"
```

긴 대화를 이어갈 때는 세션이, 장기 습관과 환경 재학습을 줄이는 데는 메모리가 핵심입니다. 이 둘을 함께 이해해야 Hermes가 "이전에도 같이 일해 본 에이전트"처럼 느껴집니다.
