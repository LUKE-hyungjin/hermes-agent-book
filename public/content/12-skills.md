# 12. 스킬

> 스킬은 에이전트가 필요할 때만 펼쳐 보는 지식 문서이자, 재사용 가능한 작업 절차 패키지입니다.

## 스킬 시스템이란?

Hermes의 스킬은 "항상 프롬프트에 들어 있는 거대한 시스템 규칙"이 아닙니다. 평소에는 목록과 설명만 가볍게 보이고, 실제로 필요할 때만 본문을 로드하는 점진적 공개(progressive disclosure) 방식으로 동작합니다. 덕분에 스킬이 많아져도 기본 세션 토큰 부담이 커지지 않습니다.

모든 스킬의 실제 저장 위치는 `~/.hermes/skills/`입니다. 번들 스킬, 허브에서 설치한 스킬, 에이전트가 직접 만든 스킬도 이 디렉터리를 중심으로 관리됩니다.

## 어떻게 사용하나요?

설치된 스킬은 자동으로 슬래시 명령이 됩니다. CLI뿐 아니라 Telegram 같은 메시징 플랫폼에서도 같은 방식으로 부를 수 있습니다.

```text
/plan 인증 제공자 마이그레이션 롤아웃 계획을 작성해줘
/github-pr-workflow auth 리팩터링용 draft PR 열어줘
/excalidraw
```

자연어로도 사용할 수 있습니다.

```bash
hermes chat --toolsets skills -q "지금 설치된 스킬 목록을 보여줘"
hermes chat --toolsets skills -q "axolotl 스킬 내용을 설명해줘"
```

특히 번들 `plan` 스킬은 "즉시 실행" 대신 구현 계획을 마크다운으로 써서 `.hermes/plans/` 아래에 저장하는 대표 예시입니다.

## 점진적 공개 구조

스킬은 대략 세 단계로 로드됩니다.

| 단계 | 동작 | 목적 |
|------|------|------|
| Level 0 | `skills_list()` | 이름, 설명, 카테고리만 확인 |
| Level 1 | `skill_view(name)` | 특정 스킬 본문 전체 로드 |
| Level 2 | `skill_view(name, path)` | 필요한 참고 파일만 추가 로드 |

이 구조의 핵심은 "모든 스킬을 매번 다 읽지 않는다"는 점입니다. 실제로 필요한 절차만 펼쳐 보기 때문에 확장성이 좋습니다.

## `SKILL.md` 형식

스킬은 보통 `SKILL.md` 하나와 선택적 참조 파일들로 구성됩니다.

```md
---
name: my-skill
description: 배포 전 체크리스트를 수행하는 스킬
version: 1.0.0
platforms: [macos, linux]
metadata:
  hermes:
    category: devops
    tags: [deploy, checklist]
    requires_toolsets: [terminal]
---

# 배포 점검 스킬

## When to Use
- 배포 직전 회귀 확인이 필요할 때

## Procedure
1. 테스트를 실행한다.
2. 설정 차이를 점검한다.
3. 배포 위험을 요약한다.
```

스킬은 단순 프롬프트 조각이 아니라, 언제 쓰는지와 절차, 함정, 검증 방법까지 함께 담는 문서로 설계하는 편이 좋습니다.

## 플랫폼 제한과 조건부 활성화

Hermes는 스킬을 항상 다 노출하지 않습니다. 운영체제나 현재 세션의 도구 가용성에 따라 숨길 수 있습니다.

| 필드 | 의미 |
|------|------|
| `platforms` | 특정 OS에서만 스킬 표시 |
| `requires_toolsets` | 특정 도구셋이 있을 때만 표시 |
| `fallback_for_toolsets` | 특정 도구셋이 없을 때만 표시 |
| `requires_tools` | 특정 개별 도구가 있을 때만 표시 |
| `fallback_for_tools` | 특정 개별 도구가 없을 때만 표시 |

```yaml
metadata:
  hermes:
    requires_toolsets: [terminal]
    fallback_for_toolsets: [web]
    requires_tools: [terminal]
```

예를 들어 유료 웹 검색 도구가 없을 때만 나타나는 대체 스킬, 혹은 macOS에서만 보이는 Apple 생태계 스킬을 만들 수 있습니다.

## 좋은 스킬의 기준

| 좋은 스킬 | 좋지 않은 스킬 |
|-----------|----------------|
| 재사용되는 절차를 묶음 | 한 번 쓰고 끝나는 임시 지시 |
| 도구 전제 조건이 명확함 | 어떤 환경에서 쓰는지 불분명 |
| 실패 모드와 검증법 포함 | 결과 판정 기준이 없음 |
| 짧고 집중된 범위 | 너무 많은 주제를 한 파일에 혼합 |

:::tip
처음부터 거대한 자동화 스킬을 만들기보다, "반복해서 다시 설명하는 절차" 하나를 먼저 스킬로 빼는 편이 좋습니다. 예를 들어 코드 리뷰 요약, 배포 전 점검, PR 초안 생성처럼 경계가 분명한 작업이 가장 적합합니다.
:::

## 스킬과 도구의 차이

도구는 에이전트가 호출하는 함수이고, 스킬은 그 도구를 언제 어떤 순서로 쓰는지 알려 주는 작업 지식입니다. 즉, 도구가 "손"이라면 스킬은 "익숙한 손동작"에 가깝습니다. Hermes를 오래 쓸수록 이 차이가 중요해집니다. 도구를 많이 아는 것보다, 어떤 절차를 스킬로 굳혀 두었는지가 실제 생산성을 크게 좌우하기 때문입니다.
