# 09. 보안

> Hermes는 "에이전트가 할 수 있는 일"을 넓히면서도, 승인 흐름과 격리 경계로 위험을 줄이는 방어 심층형 보안 모델을 사용합니다.

## Hermes 보안 모델의 핵심

Hermes는 단순히 "위험한 명령은 막는다" 수준에서 끝나지 않습니다. 누가 에이전트와 대화할 수 있는지, 어떤 명령이 사람 승인 없이는 실행되면 안 되는지, 터미널이 호스트를 직접 만지는지 컨테이너에서 격리되는지, MCP 서버에 어떤 자격 증명이 전달되는지, 프로젝트 안의 컨텍스트 파일에 프롬프트 인젝션 흔적이 없는지까지 여러 층으로 나눠 생각합니다.

| 보안 계층 | 막으려는 위험 | 대표 수단 |
|-----------|----------------|-----------|
| 사용자 승인 | 아무나 봇 사용 | allowlist, DM pairing |
| 위험 명령 통제 | 삭제/포맷/서비스 중단 | 승인 프롬프트 |
| 실행 격리 | 호스트 훼손 | Docker, Singularity, Modal 등 |
| 자격 증명 분리 | 과도한 권한 노출 | MCP 환경 변수 필터링 |
| 컨텍스트 스캔 | 프롬프트 오염 | AGENTS.md 등 스캔 |

## 위험 명령 승인

Hermes는 명령 실행 전에 위험 패턴 목록을 검사합니다. `rm -r`, 루트 경로 삭제, `chmod 777`, `mkfs`, `dd if=`, `DROP TABLE`, `DELETE FROM` without `WHERE`, `curl ... | sh`, `find -delete`, `killall` 같은 파괴적 패턴이 대표적입니다. 이런 패턴에 걸리면 CLI에서는 즉시 승인 프롬프트가 뜹니다.

```text
⚠️  DANGEROUS COMMAND: recursive delete
rm -rf /tmp/old-project
[o]nce | [s]ession | [a]lways | [d]eny
```

| 선택지 | 의미 | 저장 여부 |
|--------|------|-----------|
| `once` | 이번 한 번만 허용 | 저장 안 함 |
| `session` | 현재 세션 동안 허용 | 세션 종료 시 사라짐 |
| `always` | 영구 허용 목록 추가 | `config.yaml`에 저장 |
| `deny` | 차단 | 실행 안 함 |

메시징 게이트웨이에서는 같은 맥락이 채팅으로 전달되며, 사용자가 `yes`, `approve`, `ok` 같은 답으로 승인하거나 `no`, `deny`, `cancel`로 거절합니다. 게이트웨이 실행 시 `HERMES_EXEC_ASK=1`이 자동으로 켜지는 이유가 바로 이것입니다.

## 영구 허용 목록

반복적으로 필요한 파괴 명령이 있다면 영구 허용 목록에 저장할 수 있습니다.

```yaml
command_allowlist:
  - rm
  - systemctl
```

다만 이것은 "모든 명령을 풀어 준다"는 의미가 아니라, 해당 패턴을 이후 세션에서 조용히 승인한다는 뜻입니다. 따라서 운영 환경에서는 허용 목록을 최소화하는 편이 좋습니다.

:::tip
`always`를 누르기 전에 정말 반복 승인할 가치가 있는 패턴인지 생각하세요. 개발용 임시 디렉터리를 지우는 패턴과 운영 DB를 다루는 패턴은 같은 수준으로 다루면 안 됩니다.
:::

## 메시징 사용자 승인

게이트웨이는 기본적으로 아무에게도 열려 있지 않습니다. Telegram, Discord, Slack 같은 플랫폼에서 봇을 켠 뒤 allowlist를 구성하지 않으면 안전하게 거부하는 것이 기본값입니다.

| 검사 순서 | 예시 |
|----------|------|
| 플랫폼별 allow-all | `DISCORD_ALLOW_ALL_USERS=true` |
| DM 페어링 승인 사용자 | 페어링 코드로 승인된 사용자 |
| 플랫폼별 허용 목록 | `TELEGRAM_ALLOWED_USERS=...` |
| 전역 허용 목록 | `GATEWAY_ALLOWED_USERS=...` |
| 전역 allow-all | `GATEWAY_ALLOW_ALL_USERS=true` |
| 기본 동작 | 거부 |

```bash
TELEGRAM_ALLOWED_USERS=123456789,987654321
DISCORD_ALLOWED_USERS=111222333444555666
SLACK_ALLOWED_USERS=U01ABC123
GATEWAY_ALLOWED_USERS=123456789
```

특히 Telegram이나 Discord 봇은 터미널 명령 실행 능력까지 가지기 때문에, "일단 공개하고 나중에 막자"는 접근이 위험합니다.

## 컨테이너와 실행 격리

Hermes가 `docker`, `singularity`, `modal`, `daytona` 같은 컨테이너 백엔드에서 실행될 때는 컨테이너 자체가 주요 보안 경계가 됩니다. 이 경우 위험 명령 검사가 일부 생략될 수 있는데, 이유는 파괴적 명령이 호스트가 아니라 격리된 파일시스템 안에서만 영향을 주기 때문입니다. 반대로 `local` 백엔드에서는 호스트를 직접 건드리므로 승인 흐름의 의미가 훨씬 커집니다.

## 컨텍스트 파일과 프롬프트 인젝션

Hermes는 `AGENTS.md`, `.hermes.md`, `CLAUDE.md`, `.cursorrules`, `SOUL.md` 같은 파일을 읽어 프롬프트에 반영합니다. 이때 내용을 그대로 믿지 않고 스캔과 절단을 거칩니다. 즉, 저장소 안에 "이전 지시를 무시하라" 같은 문장이 있더라도 최소한의 방어선은 존재합니다. 하지만 이 스캔은 만능이 아니므로, 외부 저장소를 신뢰 없이 열 때는 컨텍스트 파일 자체를 사람이 검토하는 습관이 필요합니다.

## 안전 운용 체크리스트

| 항목 | 권장값 |
|------|--------|
| 개인 CLI | `local` 또는 `docker`, 허용 목록 최소화 |
| 팀 메시징 봇 | 플랫폼별 allowlist 필수 |
| 민감 서버 작업 | `ssh` 또는 컨테이너 격리 우선 |
| 외부 MCP 사용 | 필요한 헤더와 env만 노출 |
| 프로젝트 컨텍스트 | AGENTS.md 리뷰 후 사용 |

Hermes의 보안은 "절대 실행하지 않음"이 아니라 "실행하되 경계와 승인 지점을 명확히 둠"에 가깝습니다. 따라서 보안을 높이고 싶다면 기능을 끄기보다, 승인 정책과 실행 백엔드와 사용자 허용 범위를 더 엄격하게 잡는 것이 정석입니다.
