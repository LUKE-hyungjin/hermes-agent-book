# 13. 성격 설정

> Hermes의 기본 성격은 `SOUL.md`가 결정하며, 이 파일은 시스템 프롬프트의 가장 앞자리에서 에이전트의 정체성으로 작동합니다.

## `SOUL.md`가 왜 중요한가요?

Hermes는 단순히 "응답 문체 옵션" 몇 개를 켜는 방식이 아니라, `SOUL.md`라는 실제 텍스트 파일로 에이전트의 성격을 정의합니다. 이 파일은 시스템 프롬프트의 1번 슬롯, 즉 가장 핵심적인 정체성 위치에 들어갑니다. 그래서 말투, 직접성, 설명 깊이, 반박 태도, 불확실성 처리 방식 같은 기본 상호작용 품질이 여기에 의해 결정됩니다.

| 요소 | 역할 |
|------|------|
| `SOUL.md` | 지속적인 기본 성격 |
| `/personality` 프리셋 | 세션 단위 오버레이 |
| `AGENTS.md` | 프로젝트별 작업 규칙 |

만약 "Hermes가 누구인가"를 바꾸고 싶다면 `AGENTS.md`가 아니라 `SOUL.md`를 수정해야 합니다.

## 위치와 로딩 규칙

기본 위치는 아래와 같습니다.

```text
~/.hermes/SOUL.md
```

커스텀 홈 디렉터리를 쓴다면 다음 경로를 사용합니다.

```text
$HERMES_HOME/SOUL.md
```

중요한 동작 규칙은 다음과 같습니다.

| 규칙 | 의미 |
|------|------|
| 자동 시드 | 파일이 없으면 기본 `SOUL.md`를 생성 |
| 덮어쓰기 안 함 | 기존 사용자 파일은 보존 |
| 홈 디렉터리 전용 | 작업 디렉터리에서 찾지 않음 |
| 비어 있으면 폴백 | 기본 내장 정체성 사용 |
| 내용이 있으면 그대로 삽입 | 스캔과 절단 후 verbatim 주입 |

이 설계 덕분에 프로젝트마다 성격이 흔들리지 않고, 같은 Hermes 인스턴스는 언제나 같은 기본 태도를 유지합니다.

## 무엇을 넣어야 하나요?

`SOUL.md`에는 장기적으로 유지될 상호작용 원칙을 넣는 것이 좋습니다.

| 넣기 좋은 내용 | 넣지 말아야 할 내용 |
|----------------|--------------------|
| 말투와 직접성 | 프로젝트 경로 |
| 설명 길이 선호 | 특정 저장소 규칙 |
| 반박/불확실성 처리 | 오늘만 유효한 임시 지시 |
| 기술적 태도 | 파일별 작업 절차 |

즉, "`간결하게 답하라`", "`근거 없이 동의하지 마라`", "`실용성을 우선하라`"는 좋지만, "`frontend는 pnpm을 써라`"는 `AGENTS.md`에 있어야 합니다.

## 예시

아래는 원문 문서의 방향을 한국어 사용자 관점으로 옮긴 예시입니다.

```md
# Personality

You are a pragmatic senior engineer with strong taste.
You optimize for truth, clarity, and usefulness over politeness theater.

## Style
- Be direct without being cold
- Prefer substance over filler
- Admit uncertainty plainly
- Keep explanations compact unless depth is useful

## What to avoid
- Sycophancy
- Hype language
- Overexplaining obvious things
```

이런 파일은 문체만 꾸미는 것이 아니라, 답변의 밀도와 반응 태도까지 바꿉니다.

## 프롬프트 주입 방식

`SOUL.md` 내용은 별도의 포장 문구 없이 시스템 프롬프트 정체성 슬롯에 직접 들어갑니다. 단, 그대로 무조건 넣는 것은 아니고, 보안 스캔과 길이 절단을 거칩니다. 파일이 비어 있거나 읽을 수 없으면 Hermes는 내장 기본 정체성으로 폴백합니다.

## 실전 운영 팁

초기에는 완벽한 성격 파일을 만들려고 하기보다, 아래 정도만 먼저 적어도 충분합니다.

```md
# Personality
- 한국어 우선
- 군더더기 없는 설명
- 불확실하면 추측 대신 명시적으로 말하기
- 코드 변경 시 검증 결과를 함께 보고하기
```

이후 실제 사용 중 "답이 너무 장황하다", "무조건 동의하려 든다", "위험한 아이디어에 너무 순응적이다" 같은 문제를 느끼면 `SOUL.md`를 조금씩 다듬는 방식이 가장 효과적입니다.

:::tip
성격 파일은 길수록 좋은 것이 아닙니다. 모델이 항상 지켜야 할 몇 가지 강한 원칙만 남기는 편이 오히려 일관성이 좋습니다. 세세한 작업 지시는 `AGENTS.md`나 개별 세션 요청으로 분리하세요.
:::

결국 `SOUL.md`는 Hermes를 "일회성 챗봇"이 아니라 "같은 성격으로 계속 함께 일하는 에이전트"로 느끼게 만드는 핵심 장치입니다.
