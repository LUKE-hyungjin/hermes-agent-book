# 19. Discord 설정

> Discord에서 Hermes는 단순 웹훅이 아니라 전체 메시징 게이트웨이를 타는 봇으로 동작합니다. 그래서 멘션 규칙, 채널별 세션 분리, 동시성 정책을 이해하는 것이 설정 그 자체만큼 중요합니다.

## Discord에서 Hermes가 어떻게 보이는가

공식 문서는 먼저 "서버에 넣었을 때 Hermes가 실제로 어떻게 반응하는지"부터 설명합니다. 기본 동작은 다음과 같습니다.

| 맥락 | 동작 |
|------|------|
| DM | 모든 메시지에 응답, 멘션 불필요 |
| 서버 채널 | 기본적으로 `@mention`이 있어야 응답 |
| 자유 응답 채널 | `DISCORD_FREE_RESPONSE_CHANNELS`로 멘션 없이 응답 가능 |
| 스레드 | 같은 스레드 안에서 응답, 부모 채널과 세션 분리 |
| 공유 채널 | 기본적으로 사용자별 세션 분리 |

즉, `#research` 같은 공개 채널에 봇을 넣었다고 해서 모든 대화에 끼어드는 구조가 아닙니다. 기본값은 보수적이고, 필요한 채널만 자유 응답으로 여는 방식입니다.

## Discord 게이트웨이 모델

Discord의 Hermes는 상태 없는 응답기가 아닙니다. 각 메시지는 아래 순서로 처리됩니다.

1. `DISCORD_ALLOWED_USERS` 권한 확인
2. 멘션 필요 여부 또는 자유 응답 채널 여부 검사
3. 세션 키 조회
4. 기존 대화 이력 로드
5. 일반 Hermes 에이전트 실행
6. 응답을 Discord로 다시 전달

이 구조 덕분에 도구 호출, 메모리, 슬래시 명령, 음성 기능이 그대로 이어지지만, 동시에 채널 설계를 잘못하면 문맥이 복잡해질 수 있습니다.

## 세션 모델의 핵심

기본값은 `group_sessions_per_user: true`입니다.

```yaml
group_sessions_per_user: true
```

이 설정에서는 같은 채널에 Alice와 Bob이 함께 있어도 서로 다른 세션을 씁니다. 한 사람이 긴 도구 실행을 하더라도 다른 사용자의 대화 이력을 더럽히지 않습니다.

반대로 아래처럼 바꾸면 채널 전체가 하나의 공유 세션이 됩니다.

```yaml
group_sessions_per_user: false
```

공유 세션은 협업형 방에 유용할 수 있지만 다음 비용이 생깁니다.

- 사용자들이 컨텍스트와 토큰 비용을 공유함
- 한 사람의 긴 작업이 모두의 세션을 부풀릴 수 있음
- 누군가의 후속 메시지가 다른 사람 실행을 끊거나 대기시킬 수 있음

## 인터럽트와 동시성

Hermes는 세션 키를 기준으로 실행 중인 에이전트를 추적합니다. 기본 설정에서는 사용자가 자기 세션만 끊습니다. 공유 세션 모드에서는 방 전체가 하나의 실행 슬롯을 같이 쓰게 됩니다.

:::tip
일반적인 도움말 채널을 만들고 싶다면 `DISCORD_FREE_RESPONSE_CHANNELS`를 쓰고, `group_sessions_per_user: true`는 유지하는 조합이 가장 안전합니다.
:::

## 설정 시작

문서의 첫 단계는 Discord Developer Portal에서 애플리케이션을 만드는 것입니다.

1. Discord Developer Portal에 로그인합니다.
2. `New Application`을 누릅니다.
3. 예를 들어 `Hermes Agent` 같은 이름을 넣습니다.
4. 약관에 동의하고 생성합니다.
5. 생성 후 `Application ID`를 기록합니다.

이후의 토큰 발급과 봇 권한 설정은 보통 설정 마법사로 처리하는 편이 가장 빠릅니다.

```bash
hermes gateway setup
```

## 운영에 바로 쓰는 환경 변수

문서에서 직접 언급된 Discord 관련 핵심 값은 다음과 같습니다.

```bash
DISCORD_ALLOWED_USERS=123456789012345678
DISCORD_FREE_RESPONSE_CHANNELS=research,bot-help
DISCORD_REQUIRE_MENTION=false
```

`DISCORD_REQUIRE_MENTION=false`는 전역적으로 멘션 요구를 끄는 옵션이라 편리하지만, 팀 서버에서는 채널 소음이 커질 수 있습니다. 보통은 전체 해제보다 자유 응답 채널만 지정하는 방식이 더 낫습니다.

## 어떤 식으로 쓰면 좋은가

| 운영 방식 | 추천도 | 이유 |
|------|------|------|
| DM 전용 | 높음 | 개인별 세션이 명확함 |
| `@mention` 기반 서버 채널 | 높음 | 소음이 적고 예측 가능 |
| 자유 응답 전용 봇 채널 | 높음 | 팀 협업에 편리 |
| 채널 전체 공유 세션 | 선택적 | 협업형 방에는 유용하지만 충돌 위험 존재 |

Discord 설정의 포인트는 "봇을 띄운다"가 아니라 "서버 안에서 어떤 문맥 경계를 만들 것인가"입니다. 이 부분만 제대로 잡으면 Telegram보다 훨씬 강한 협업 인터페이스가 됩니다.
