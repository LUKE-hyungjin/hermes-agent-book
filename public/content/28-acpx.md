# 28. acpx: 에이전트 간 통신

이 문서는 **Hermes 공식 핵심 기능 문서라기보다 생태계 참고용 보조 장**입니다. `acpx` 자체는 OpenClaw 쪽에서 공개한 헤드리스 ACP 클라이언트이며, Hermes와의 구체적 연결 방식은 버전과 구성에 따라 달라질 수 있습니다.

:::tip
초보자는 이 장을 당장 실습 대상으로 보기보다, "Hermes가 필요하면 다른 에이전트에게도 일을 넘길 수 있다"는 개념 참고용으로 읽으면 충분합니다.
:::

## acpx란?

`acpx`는 Agent Client Protocol(ACP)을 위한 CLI 클라이언트입니다. 어렵게 보기보다, Hermes가 다른 코딩 에이전트에게 일을 보내고 결과를 다시 받게 해 주는 연결 도구라고 이해하면 가장 쉽습니다.

Hermes 관점에서 보면 `acpx`는 "외부 에이전트 호출기"에 가깝습니다. Hermes가 직접 모든 일을 하지 않고, 필요할 때 Codex나 Claude Code 같은 다른 에이전트에게 작업을 위임하고 결과를 다시 받아오게 만드는 연결 다리로 이해하면 됩니다.

## 왜 중요한가?

- PTY 문자 출력 파싱보다 구조화된 통신에 가깝습니다.
- 세션을 유지하면서 여러 턴을 이어갈 수 있습니다.
- 이름 있는 세션을 만들어 병렬 작업 흐름을 나누기 쉽습니다.
- 취소, 상태 확인, 큐잉 같은 제어가 비교적 명확합니다.
- 대상이 ACP를 네이티브로 말하든, 어댑터를 통해 말하든 같은 CLI 표면으로 다룰 수 있습니다.

특히 Hermes처럼 장기 실행형 에이전트와 함께 쓰면, "현재 작업은 외부 코딩 에이전트에 넘기고, Hermes는 상위 맥락과 장기 기억을 유지하는" 식의 역할 분리가 쉬워집니다.

## 지원 대상

공개 저장소와 문서에서 바로 확인되는 대상 중, 이 문서에서 중요한 것은 다음 세 가지입니다.

| 대상 | `acpx`에서의 이름 | 연결 방식 | 해석 포인트 |
| --- | --- | --- | --- |
| Codex | `codex` | 어댑터 래핑 | 코드 수정, 테스트, 리포지터리 작업을 위임하는 용도로 이해하기 쉽습니다 |
| Claude Code | `claude` | 어댑터 래핑 | Anthropic 계열 코딩 워크플로를 붙이고 싶을 때 자연스럽습니다 |
| OpenClaw | `openclaw` | `openclaw acp` 네이티브 브리지 | 메신저/게이트웨이 쪽 세션과 연결할 때 특히 의미가 있습니다 |

정확히 말하면 `acpx`는 이 셋 외에도 더 많은 대상에 대응할 수 있도록 설계되어 있습니다. 다만 여기서는 Hermes와 함께 설명할 때 자주 거론되는 Codex, Claude Code, OpenClaw만 다룹니다.

## 예시 워크플로

다음은 개념적으로 이해하기 쉬운 흐름입니다.

1. Hermes가 상위 목표를 받습니다.
2. 작업 성격을 보고 적절한 외부 대상을 고릅니다.
3. `acpx`가 ACP 세션을 만들거나 기존 세션에 붙습니다.
4. 외부 에이전트가 분석, 수정, 요약 같은 작업을 수행합니다.
5. Hermes가 결과를 받아 장기 맥락에 반영하고 다음 행동을 결정합니다.

예를 들면 이런 식으로 개념적으로 이해할 수 있습니다. 아래 명령은 버전에 따라 달라질 수 있는 예시이므로, 실제 사용 전에는 upstream `acpx` 문서를 함께 확인하는 편이 안전합니다.

```text
Hermes가 목표를 받음
→ acpx가 적절한 외부 에이전트 세션에 연결
→ 외부 에이전트가 분석/수정/리뷰 수행
→ 결과가 다시 Hermes로 돌아옴
```

이 예시는 "Hermes가 외부 에이전트에게 일을 보낸다"는 개념을 보여주기 위한 것입니다. 실제 래핑 방식은 Hermes 쪽 통합 레이어가 어떻게 구현되었는지에 따라 달라질 수 있습니다.

## 실무 메모

- `acpx`는 공개 README에서 알파(alpha) 상태라고 밝히고 있으므로, CLI와 런타임 인터페이스가 바뀔 가능성을 염두에 두는 편이 좋습니다.
- 대상마다 ACP 지원 수준이 다를 수 있습니다. 같은 `acpx` 명령을 써도 제공되는 이벤트, 도구 스트리밍, 설정 옵션이 완전히 같다고 단정하면 안 됩니다.
- OpenClaw의 `openclaw acp`는 문서상 "브리지"로 설명되며, 일부 ACP 기능은 부분 지원 또는 미지원입니다.
- 비대화형 환경에서는 권한 승인 정책을 미리 정해야 합니다. 자동 승인 범위를 넓히면 편하지만 위험도도 함께 커집니다.
- 장기 작업은 이름 있는 세션으로 분리하는 편이 실용적입니다. 예를 들어 `backend`, `docs`, `review`처럼 나누면 Hermes가 후속 호출을 관리하기 쉽습니다.

## 대상별 강점

### Codex

- 코드 수정, 테스트 실행, 리포지터리 단위 작업에 초점을 맞추기 좋습니다.
- "버그 고치기", "타입 오류 정리", "패치 적용 후 검증" 같은 개발 작업에 잘 맞습니다.
- Hermes가 상위 오케스트레이션을 맡고, 실제 구현 턴은 Codex에 넘기는 패턴이 자연스럽습니다.

### Claude Code

- 코드 설명, 리팩터링, 구현 보조, 워크플로 자동화 같은 터미널 중심 개발 흐름에 강점이 있습니다.
- 이미 Anthropic 생태계를 쓰고 있다면 연결 맥락이 비교적 명확합니다.
- 구현 자체뿐 아니라 코드베이스 탐색과 설명 업무를 나눠 맡기기에도 적합합니다.

### OpenClaw

- 단순 코딩 에이전트라기보다, 게이트웨이와 세션 허브라는 성격이 중요합니다.
- Hermes가 외부 메시징 세션이나 장기 대화 흐름과 연결해야 할 때 더 의미가 큽니다.
- 즉, "코드를 직접 고치는 능력"만 보기보다 "어디에서든 이어지는 세션"이 필요한 상황에서 강점을 봐야 합니다.
